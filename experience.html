AirSense IoT Platform

Founder — Software & Full Stack
2025

Summary

End-to-end IoT platform for real-time air quality measurement, analysis and emergency alerts. I own and build the software architecture, backend, mobile app and data layer; field hardware and the physical layer progress with team collaboration. I securely collect sensor data, process it in a centralised database and present it through a mobile application in a readable, actionable format.

Problem Statement

Scattered, manual and delayed management of data from air quality sensors at different locations; inability to receive notifications when instantaneous thresholds are exceeded; and historical data not being kept in a format suitable for analytics.

System Architecture

A layered architecture was established: ESP32-based sensors at the device layer, MQTT-based messaging at the communication layer, Python services and a relational database at the backend layer, and a React Native mobile application at the client layer. The API surface was designed according to REST principles, using separate endpoints for read/write operations and a simple, versionable endpoint structure.

Implementation Details

Backend

Python-written services on the backend side consume data from the MQTT broker and, after validation and normalisation, write it to a SQL-based database. Data stored in tables indexed by timestamp, device ID and sensor type is served on the read side through lightweight SELECT queries and, where necessary, aggregations (GROUP BY, window functions).

Frontend

In the React Native mobile application, screens were kept modular with a component-based structure. Periodic polling and a lightweight cache layer were used for real-time data reading. List and chart components were designed with simple state management to re-render with low latency.

Database

The database was established with a normalised schema that splits devices, measurements and threshold definitions into separate tables. Composite indexes on timestamp and device ID were used for read-heavy queries. WHERE conditions were structured to be index-friendly for frequently queried scenarios such as the last X records.

Performance & Optimisation

Thanks to indexes created on timestamp and device ID, response times in reporting queries were visibly improved. By selecting only the required fields and applying pagination, the mobile application's data-fetch time and payload size were reduced.

Engineering Practices

The layered structure provided separation close to SOLID principles by decoupling domain-specific business rules from the data access layer. Configuration, environment variables and reusable helper functions were isolated into separate modules. Modular functions were preferred for sensor data processing, error tolerance and logging.

MixCodder Inventory ERP

Project Manager, Backend & Database Manager
2025

Summary

Inventory-focused ERP system for SMEs. I consolidate inventory, order and reporting processes under a central model using modules running on different platforms (C# ASP.NET Core and Python Django).

Problem Statement

Manually tracked inventory movements, inconsistent reports and data duplicated across different systems led to both operational errors and incorrect metrics in the decision-making process.

System Architecture

The system was approached with a typical layered architecture: presentation layer (web interface), business rules layer (domain services) and data access layer (repository/ORM). The ASP.NET Core side provides REST-style APIs; the Django side provides an admin panel and certain domain-specific functions. The focus was on maintaining data consistency between the two platforms using a shared domain model.

Implementation Details

Backend

In ASP.NET Core, the controller layer was kept simple, only receiving HTTP requests and delegating to application/domain services for business rules. ORM was used for database access alongside raw SQL queries where needed. CRUD operations were accelerated through model definitions and the admin panel in Django.

Frontend

The web interface was designed with table and card components to make inventory lists and movement details readable. Filtering, sorting and pagination functions improved usability across large datasets.

Database

Concepts such as inventory, movements, warehouses and users were modelled in separate tables with a relational (foreign key) structure. Indexes were defined on date, stock and warehouse columns for frequently used reports. Transaction structures ensured consistency in critical movements.

Performance & Optimisation

Query plans were examined and indexes were added to the necessary fields; heavy reports were written with a filter-first, then group logic. Only the required columns were selected and unnecessary joins were simplified to reduce response times.

Engineering Practices

A structure close to SOLID principles using dependency injection was preferred. Common code pieces were gathered under helper services and extension methods. Contracts between layers (DTOs, view models) isolated the domain model from the UI.

DGPCA (NVI), Ministry of Interior — BVYS Software Branch

Intern — Software Branch (BVYS)
2025–2026 (Internship)

Summary

At the General Directorate of Population and Citizenship Affairs (NVI), Ministry of Interior of the Republic of Türkiye, I interned with the Biometric Data Management System (BVYS) Software Branch. My work covered advanced PostgreSQL operations on live production and test systems, enterprise Linux infrastructure, cross-system automation, and engineering on Turkey’s national biometric data platform—including fingerprint recognition and biometric processing pipelines supported by dedicated technical training. I also contributed to end-to-end test automation architecture using C# Selenium and Playwright. I used WEB-AUDIT and WEB-MONITOR—tools I built—in this internship as well (documented separately on this page). Because BVYS is a public-sector programme with closed-source and operational-security constraints, I cannot provide a general or detailed public explanation of internal system design, data flows, or proprietary components here; the sections below describe my responsibilities, tooling, and practices at an appropriate level of abstraction.

Problem Statement

National-scale biometric and citizen-services workloads require databases and infrastructure that stay reliable under heavy read/write traffic, strict audit expectations, and continuous operational change. Manual DBA and integration steps did not scale; reporting and maintenance queries on large datasets produced avoidable I/O; and web-facing services needed repeatable quality and security signals beyond ad hoc manual checks. The organisation also needed alignment with enterprise and defence-grade stability practices while keeping automation auditable and on-prem.

System Architecture

Day-to-day work sat at the intersection of four layers: a central PostgreSQL data tier for identity, biometric, and transaction data; Ubuntu-based server and cluster infrastructure observed through Kubernetes and Zabbix; an automation tier where advanced SQL and Bash scripts orchestrate cross-system data movement without manual hand-offs; and a quality layer where browser automation (Playwright, Selenium) and TypeScript-backed audit/monitor tooling produce continuous signals for web applications. NVI operational standards were complemented by mentorship from Havelsan Linux engineers on military/enterprise stability processes—change control, hardening, and production discipline—rather than generic desktop Linux administration.

Implementation Details

Backend

On infrastructure and DevOps, I worked hands-on with Ubuntu server administration, Kubernetes workloads, and Zabbix-based monitoring in an enterprise public-sector environment. Under guidance from Havelsan Linux engineers, I applied military/enterprise stability practices to patching, service health, incident response, and safe change windows. I developed advanced SQL scripts to replace repetitive manual DBA and integration tasks, and embedded complex SQL inside Linux Bash (.sh) jobs so data could flow across systems with scheduled, auditable automation.

Frontend

On the BVYS application side, I received specialised training and contributed to the Biometric Data Management System—fingerprint recognition and broader biometric data processing—in active project architectures deployed to the field. Work emphasised live-system behaviour, traceability, and coordination with application teams rather than public UI portfolio deliverables.

Database

Database management was a core responsibility on both production and large-scale test instances. I operated PostgreSQL at an advanced level via terminal (psql, Bash) and GUI (pgAdmin4): schema and query tuning, index design, EXPLAIN/EXPLAIN ANALYZE-driven optimisation, and hands-on support for active project architectures in the field. Optimisation cycles followed measure → hypothesise → controlled change → re-measure, with B-tree indexes and query rewrites targeted at patterns that caused sequential scans and reporting I/O spikes on high-volume biometric datasets.

Performance & Optimisation

Targeted index and query revisions shortened critical paths on large biometric tables and reduced unnecessary full scans and reporting I/O. Infrastructure monitoring and scripted automation cut manual turnaround for recurring operational tasks. Parallel browser-based audit and HTTP/HTTPS monitor runs (see WEB-AUDIT and WEB-MONITOR case studies) improved visibility into latency, availability, and security posture without relying on external SaaS.

Engineering Practices

Engineering practice blended public-sector rigour with Havelsan-informed enterprise stability: documented changes, reproducible scripts, defense-in-depth around production access, and clear separation of test vs production work. Test automation was treated as platform engineering—C# Selenium and Playwright for E2E coverage, TypeScript-supported WEB-AUDIT/WEB-MONITOR for continuous web performance and security tracking—integrated into BVYS delivery culture rather than one-off manual QA. Closed-source obligations were respected: no internal architecture or sensitive implementation detail is published on this site.

WEB-MONITOR – Self-Hosted HTTP/HTTPS Monitoring Platform

Project Owner & Developer
2025–2026

Summary

WEB-MONITOR is a self-hosted, asynchronous HTTP/HTTPS monitoring platform I built and own. It lets teams track web application performance entirely on-prem, with continuous uptime, latency, and SLO visibility through a central dashboard, Prometheus metrics, and CLI reports. TypeScript supports tooling and integrations alongside the Python/FastAPI core. [Source: `WEB-MONITOR` README](https://github.com/ArdaKaradag36/WEB-MONITOR)

Problem Statement

Enterprise web services need performance and availability tracking that does not depend on third-party SaaS, can run in sovereign data centres, and produces signals operators can act on during incidents. Ping-only checks miss latency regressions and SLO breaches; scattered targets and global outages often trigger alert storms without backpressure or quality metrics.

System Architecture

The system is built around an async monitor loop that reads target definitions from YAML/TXT files, a central SQLite database running in WAL mode, a FastAPI‑based REST API and NOC dashboard, a Prometheus `/metrics` endpoint, and CLI/TUI reporting. Nginx sits in front as a reverse proxy, protecting sensitive endpoints like `/metrics` with Basic Auth and network policy. With Docker Compose, the full stack can be started with a single command.

Implementation Details

Backend

Using Python 3.10+, asyncio and aiohttp, we implemented a monitor loop that can probe thousands of HTTP/HTTPS targets concurrently with AIMD‑style concurrency and backpressure control. To harden against SSRF, DNS resolution and private IP filtering were added, alongside response body size limits and per‑target TLS certificate expiry checks that are configurable via YAML. Health, incident and SLO calculation logic is exposed through FastAPI routes.

Frontend

For the FastAPI dashboard, I worked with a lightweight NOC UI built using HTML, CSS and Vanilla JavaScript. Service status summaries, incident lists and SLO reports are pulled from the API as JSON and rendered as responsive cards and tables optimised for both desktop and mobile. The focus is on clear incident timelines and quick drill‑down into latency distributions.

Database

SQLite is configured in WAL mode to balance concurrent writes from the async monitor loop with read traffic from the API and dashboard. Uptime, latency and incident records are stored in indexed tables keyed by target and wave (monitor iteration). Percentile calculations (p50/p95/p99) are implemented with queries that carefully trade off read performance against write cost.

Performance & Optimisation

Thanks to the async I/O model and AIMD backpressure, WEB-MONITOR can scale to a high number of targets while remaining stable. HTTP timeout, retry, jitter and a global circuit breaker help prevent alert storms during network incidents. Prometheus metrics provide visibility into latency distributions and error budget consumption, guiding further tuning of queries, indexes and concurrency limits.

Engineering Practices

Practices include environment-based configuration, defense-in-depth against SSRF and secret leakage, and CI with Ruff, Pytest, and Gitleaks. Docker Compose and START/README documentation support reproducible on-prem deployment. It is designed for public-sector and closed environments; setup and architecture details are in the README.

WEB-AUDIT – Browser-Based Web Application Auditing Tool

Project Owner & Developer
2025–2026

Summary

WEB-AUDIT is a TypeScript and Playwright-based browser auditing tool I built and own. It automates technical risk discovery for web applications in closed, on-prem environments and delivers repeatable security and quality signals: headers, network failures, accessibility/SEO/privacy indicators, and UI interaction coverage as JSON artefacts for security, QA, and platform teams. [Source: `WEB-AUDIT` README](https://github.com/ArdaKaradag36/WEB-AUDIT)

Problem Statement

Enterprise and self-hosted web applications need more than uptime or manual smoke tests—teams require automated, CI-friendly scans for security posture, third-party tracking, accessibility gaps, and network errors. Ad hoc browser reviews stay slow and inconsistent, making reproducible signals hard to obtain.

System Architecture

WEB-AUDIT consists of a Playwright‑based runner core, a Postgres job store that persists audit jobs and results, queue worker services that execute and process jobs, and components that aggregate and surface outputs. SSRF protections exist on both host and runner; for enterprise hardening, features like mTLS, signing and policy enforcement are included in the design. Each run typically produces `summary.json`, `gaps.json`, `ui-inventory.json` (when available), `console.json`, `network.json`, `request_failed.json` and `run.complete.json`.

Implementation Details

Backend

In the runner core, we designed scenarios that capture HTTP/network/console events and, as pages load, analyse security headers, redirects, third‑party scripts and failed requests. SSRF exposure is reduced using host allowlists, redirect rules and additional checks around sensitive endpoints. The Postgres schema for jobs and runs is modelled so that workers can be scaled out horizontally while keeping job state consistent.

Frontend

Since the focus is automation and reporting, UI work concentrates on basic reporting views and CI‑oriented visualisations of audit outputs. Dedicated a11y jobs feed accessibility findings into separate dashboards and documentation so that product and QA teams can act on them.

Database

On Postgres we designed tables for jobs, runs, steps and artefacts. Considering artefact size and query patterns, we added indexes and retention policies that keep historical data useful without uncontrolled growth. State machines and queue semantics ensure workers can process jobs in parallel without double‑processing.

Performance & Optimisation

Parallel execution of Playwright‑based audits and the job store architecture allow multiple sites and scenarios to complete within acceptable CI time budgets. Where streaming was not necessary, batch processing of network and console logs was used to strike a balance between memory usage, throughput and artefact size.

Engineering Practices

The product stance is security-first signals—automation output informs risk and compliance workflows without claiming full business-logic proof. Enterprise hardening guidance (mTLS, signing, policies) and reproducible runner images support safe on-prem CI adoption. It is designed for public-sector and closed environments; deeper architecture lives in the README and under `docs/`.
