My AI-Assisted Development Ecosystem

Layered, deliberate AI

I route generative AI by sensitivity, compute and task — not by habit. Lightweight models on a laptop handle everyday work; a desktop tower runs large Qwen stacks for deep analysis; cloud assistants in Cursor and Gemini accelerate design and delivery. In public-sector contexts, closed, on-prem inference is not a preference but a requirement — and I build workflows that respect that boundary.

Laptop — Everyday Local AI

Small open-weight models on the laptop are my always-on, offline-capable layer — fast enough for daily rhythm without reaching for the cloud.

7B–14B class Ollama / LM Studio Offline-first
  • Daily cadence: Drafting notes, summarising meetings, sketching SQL or API shapes, and sanity-checking logic before a commit — with minimal latency and no network dependency.
  • Privacy by default: Personal drafts, early ideas and pre-classified material stay on-device until I deliberately promote them to a heavier tier.

Desktop — Large Qwen for Analysis

On a dedicated desktop (kasa) with enough VRAM, I run large Qwen models — Qwen2.5 / Qwen3 class — where depth matters more than portability.

Qwen2.5 Qwen3 32B+ Data analysis
  • Analytical workloads: Exploring tabular exports, log corpora, schema relationships and comparative metrics across long documents — without pushing sensitive datasets to a public API.
  • Reasoning at scale: Multi-file synthesis, hypothesis testing on anomalies, and structured narratives that feed architecture decisions and reporting — all on local weights.

Public Sector — Why Closed / Local AI

In government and institutional settings, generative AI must align with sovereignty and compliance — not convenience alone.

Data residency Air-gapped KVKK / GDPR On-prem
  • Sovereignty: Citizen, operational and audit data remain inside the institutional perimeter — no third-party inference trail.
  • No external leakage: Closed or on-prem stacks ensure prompts, attachments and model outputs never cross an uncontrolled boundary.
  • Compliance & procurement: Traceable deployments, classification-aware workflows, and alignment with public procurement and information-security policy — not ad-hoc cloud trials.

Gemini — Design Reasoning & Long Context

Gemini is where I lean on strong reasoning and a wide context window — especially before implementation hardens.

Gemini Pro BPMN Long-context Architecture
  • Process & architecture: Modelling complex flows in BPMN, stress-testing integration boundaries, and comparing design options while the codebase is still flexible.
  • Efficiency where it shines: Ingesting large spec folders, meeting notes and multi-document corpora in one pass — comparative technical reports and synthesis that would take hours manually, delivered in a structured, reviewable form.

Cursor — AI-Native Development

For roughly six months, Cursor has been my primary IDE — the place where AI is woven into the edit-compile-review loop, not bolted on afterward.

Cursor Agent mode Multi-model
  • Workflow integration: Inline completion, scoped refactors, test scaffolding and agent-driven tasks across files — with context from the open repository, not isolated prompts.
  • Model routing: I switch between frontier models inside Cursor by task — code review and security-sensitive diffs on one model, algorithm tuning and exploratory spikes on another — without leaving the editor.
  • Quality gate: Generated SQL, configs and tests are never merged blindly; they pass the same manual review I apply to any contribution.

Cloud Models & Services

When data classification allows, cloud frontier models extend reach — long-form writing, broad research and API-backed automation.

ChatGPT Plus GPT-4o Claude Cloud API
  • Documentation & strategy: PRDs, API contract drafts, stakeholder reports and polished technical prose — with ChatGPT Plus as a long-standing layer and Claude / GPT where reasoning depth fits.
  • Bounded use: Cloud is chosen explicitly after classifying the material; anything restricted stays on laptop, desktop Qwen, or on-prem — never “by default.”

AI Across Project Delivery Patterns

Across ERP, IoT, mobile and database work, I reuse the same patterns — generic roles for AI, not one-off tricks tied to a single codebase.

Full-stack IoT SQL / APIs Docs pipeline
  • Architecture spikes: I use AI to pressure-test API shapes, state machines and error paths before locking implementation — then discard drafts that do not survive review.
  • Embedded & integration: Datasheet interpretation, MQTT / serial debugging narratives, and hardware–software boundary checks during bring-up.
  • Data layer: Query drafts, index rationale and migration sketches — always validated against production constraints and explain plans.
  • Documentation chain: Requirements → diagrams → implementation notes, with each artefact human-owned and AI-accelerated.
